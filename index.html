Learning Git: New PC Setup

Moving to a new computer and need to reconnect with your GitHub repository? This guide walks you through all the methods to get back to coding quickly and efficiently.

Your Situation:
"I'm using my computer and working with a project connected to GitHub. I changed PCs. How do I connect again with the same repo using Git? Do I have to git clone first and then add remote, or are there other methods?"
Method 1: Clone + Remote
Method 2: Fork + Upstream
Method 3: Manual Transfer
Method 4: SSH Setup
Setup Checklist
1

Clone + Existing Remote (Most Common)

Most Common Simple Recommended

If you just want to get the code and continue where you left off, this is the simplest and most reliable method. The remote is automatically set up as origin when you clone.

# 1. Clone the repository
git clone https://github.com/username/repo-name.git

# 2. Navigate into the project
cd repo-name

# 3. Check remote (it's already set to origin)
git remote -v
# Should show: origin → your GitHub repo

# 4. Start working and push changes
git add .
git commit -m "Your message"
git push origin main
When to use this: 99% of cases - when you just need to get your code on a new machine and continue working.

How This Works

  • git clone downloads the entire repository to your new PC
  • Automatically creates a remote called origin pointing to the GitHub repo
  • You get all commit history and the current state of the repository
  • You can immediately start working and pushing changes
Note about branches: After cloning, you only get the default branch (usually main/master). To get other branches, use:

git checkout -b branch-name origin/branch-name
2

Clone + Set Upstream (If Forked)

Forked Repos Open Source

If you're working with a forked repository (common in open source contributions), you need to set up two remotes: one for your fork and one for the original repository.

# 1. Clone your fork
git clone https://github.com/YOUR-USERNAME/repo-name.git

# 2. Add upstream (original repo)
git remote add upstream https://github.com/ORIGINAL-OWNER/repo-name.git

# 3. Verify remotes
git remote -v
# origin → your fork
# upstream → original repo
Why two remotes? origin is your fork where you push changes. upstream is the original repo to sync with latest changes.

Working with Forks

With this setup, you can:

  • Push your changes to your fork (origin)
  • Pull updates from the original repo (upstream) to stay current
  • Create pull requests from your fork to the original repository
Typical fork workflow:

# Sync with original repo
git fetch upstream
git merge upstream/main

# Make and commit changes
git add .
git commit -m "My changes"

# Push to your fork
git push origin main
3

Transfer Local Repository + Add Remote

Manual Transfer Advanced

If you copied files manually (via USB, cloud storage, etc.) and want to connect them to your GitHub repository. Not generally recommended but useful in specific situations.

# 1. Initialize git in your project folder
cd project-folder
git init

# 2. Add remote
git remote add origin https://github.com/username/repo-name.git

# 3. Pull the latest code first
git pull origin main --allow-unrelated-histories

# 4. If you have existing files, add and commit them
git add .
git commit -m "Initial commit from new PC"

# 5. Push (might need to force if histories differ)
git push -u origin main
Use with caution: The --allow-unrelated-histories flag is needed when combining unrelated git histories. Force push (--force) may be needed but can be dangerous.

When to Use This Method

Consider this approach only when:

  • You manually copied files without using git
  • You have uncommitted changes from your old PC that you couldn't push
  • You're recovering from a backup without git history
Better Alternative: If you have uncommitted work on your old PC, try to commit and push it first. If that's not possible, commit locally on the old PC, then clone on the new PC.
4

Using SSH (More Secure)

SSH Security No Password Prompt

SSH provides a more secure way to connect to GitHub without entering your password each time. Once set up, it's more convenient for frequent use.

# 1. Generate SSH key on new PC (if not already)
ssh-keygen -t ed25519 -C "your-email@example.com"
# Press Enter to accept defaults

# 2. Add SSH key to GitHub account
# Copy the public key:
cat ~/.ssh/id_ed25519.pub
# Add to: GitHub → Settings → SSH and GPG keys

# 3. Clone using SSH
git clone git@github.com:username/repo-name.git
SSH Benefits: More secure than HTTPS, no password prompts after setup, required for some organizations.

HTTPS vs SSH Authentication

HTTPS: You'll be prompted for username and password. If you have 2FA enabled, you need to use a Personal Access Token (PAT) as your password.

SSH: Uses cryptographic keys. The private key stays on your computer, the public key is added to GitHub. More secure and convenient once set up.

Authentication Issues with HTTPS: If using HTTPS with 2FA, generate a Personal Access Token (PAT) on GitHub and use it as your password when prompted.

Complete Setup Checklist

1. Install Git on new PC
# Check installation
git --version
2. Configure Git (once per PC)
git config --global user.name "Your Name"
git config --global user.email "your-email@example.com"
3. Set Up Authentication
• HTTPS: Use credentials or Personal Access Token (if 2FA enabled)
• SSH: Set up SSH keys as shown in Method 4
4. Get Your Code
# Most straightforward approach
git clone https://github.com/username/repo-name.git
5. Verify Setup
cd repo-name
git status
git log --oneline -5 # See recent commits
git remote -v # Verify remote URL

Recommended Workflow

For most users switching to a new PC, this simple workflow is all you need:

# Simple and effective
git clone https://github.com/username/repo.git
cd repo
# Work on your changes...
git add .
git commit -m "Changes from new PC"
git push origin main
Which Method Should You Choose?
  • 99% of cases: Use Method 1 (git clone) - it's the simplest and most reliable
  • Working with forks: Use Method 2
  • Already copied files manually: Use Method 3
  • Prefer SSH/security: Use Method 4

    • Just git clone is usually all you need. The remote is automatically set up as origin, so you can immediately start working and pushing changes.
